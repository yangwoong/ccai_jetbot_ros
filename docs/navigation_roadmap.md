# 핵심 임무 로드맵: 지역 인지 순찰

이 프로젝트의 핵심 목표를 기록합니다. YOLO와 LLM(Qwen3-VL-70B)은 이 파이프라인의 핵심 구성요소이며, 아래 우선순위 원칙에 따라 개발합니다.

## 핵심 파이프라인 (목표)

```
1. 이동 가능 바닥면적/장애물 검출 및 회피 (YOLO + 카메라)
   ↓
2. 이동 가능 지역 탐색 및 라벨링 (LLM으로 현관/복도/엘리베이터실/주방/거실 등 주변 상황 인지)
   ↓
3. 내비게이션 지도 생성
   ↓
4. 관리자로부터 순찰 지역 임무 할당
   ↓
5. 순찰하며 이상 상황 감지 및 이벤트 메시지 전송
```

## 우선순위 원칙

1. **임무 수행 안정성이 최우선입니다.** 이동 범위 탐색·장애물 회피가 끊김 없이 동작하는 것이 최신 YOLO 버전을 쓰는 것보다 중요합니다. 특정 YOLO 버전/모델이 이 Jetson(OpenCV 4.5.0, CUDA DNN)에서 불안정하면 **주저 없이 더 오래되고 검증된 버전(예: YOLOv8n → YOLOv5n, 또는 opset/입력 크기 조정)으로 낮춥니다.** `vision_nav_node`의 자동 폴백(CUDA 실패 → CPU 재시도 → 실패 시 YOLO 비활성화하고 엣지 밀도 회피 + HOG로 계속 동작, [docs/vision_and_alerts.md](vision_and_alerts.md) 참고)은 이 원칙을 코드로 구현한 것이며, 앞으로 YOLO를 확장할 때도 이 "항상 안전하게 성능이 낮은 방식으로 폴백" 구조를 유지해야 합니다.
2. **LLM(Qwen3-VL-70B)을 최대한 활용합니다.** 현재는 명령 라우팅(`llm_control_node`)과 위험 요약(`vlm_client_node`) 두 곳에만 쓰이고 있습니다. 아래 2단계(지역 라벨링), 4단계(임무 할당 시 지역명 해석), 5단계(이상 상황 설명)에도 VLM 호출을 적극적으로 추가해야 합니다. 모델은 `.env`의 `CCAI_VLLM_MODEL`로 지정하며 기본값을 `Qwen/Qwen3-VL-70B-Instruct`로 맞춰뒀습니다.

## 단계별 현재 구현 상태

| 단계 | 상태 | 담당 코드 |
|---|---|---|
| 1. 바닥면적/장애물 검출·회피 | **부분 구현** | `vision_nav_node`: 엣지 밀도 기반 주행 가능 영역 추정 + YOLO 바운딩박스 기반 장애물 검출을 결합. 다만 픽셀 단위 "주행 가능 면적" 세그멘테이션은 아직 없고, 엣지 밀도/객체 박스로 간접 추정하는 수준입니다. |
| 2. 지역 탐색 및 라벨링 (LLM) | **미구현** | 없음. `vlm_client_node`는 현재 "위험/평상시" 판단과 50자 요약만 하고, 공간 종류(현관/복도/주방 등) 분류는 하지 않습니다. **다음 작업의 핵심 공백입니다.** |
| 3. 내비게이션 지도 생성 | **미구현** | 오도메트리·IMU·SLAM이 없어 미터 단위 지도는 만들 수 없습니다. 아래 계획대로 위상(topological) 지도부터 시작할 예정입니다. |
| 4. 관리자로부터 지역 임무 할당 | **부분 구현** | `mission.py`의 `inspect <target>` 명령이 있지만 `target`은 자유 텍스트일 뿐 실제 지도의 라벨과 연결되어 있지 않습니다. 2·3단계가 있어야 의미가 생깁니다. |
| 5. 순찰 중 이상 감지 및 알림 | **구현됨** | `vlm_client_node`(RISK/NORMAL 판단) → `patrol_node`(`/ccai/events` 발행) → `telegram_bridge_node`/`web_chat_node`(관리자에게 전달). [docs/vision_and_alerts.md](vision_and_alerts.md) 참고. |

## 다음 작업 계획 (2·3·4단계를 채우는 순서)

오도메트리 없이 진짜 미터 단위 SLAM 지도를 만드는 건 이 하드웨어(휠 인코더/IMU 없음)로는 현실적이지 않습니다. 대신 **위상 지도(topological map)**로 시작하는 게 안정적입니다 — "정확한 좌표"가 아니라 "어떤 구역에서 어떤 구역으로 이동했다"는 그래프입니다.

1. **탐색 모드 추가**: `patrol_node`에 새 상태(예: `EXPLORING`)를 추가해서, 일정 거리/시간마다 정지하고 VLM에게 "이 장면이 어떤 공간인지(현관/복도/엘리베이터실/주방/거실/기타)" 분류를 요청하는 탐색 임무를 수행합니다. `vlm_client_node`에 지역 분류 전용 프롬프트/파라미터를 추가합니다(현재 위험 판단 프롬프트와 별도).
2. **위상 지도 기록**: 탐색 중 방문한 지점을 순서대로 노드로 기록하고(타임스탬프, 라벨, 대략적인 진입 방향/거리), 노드 간 이동 관계를 그래프로 저장합니다(예: `data/nav_map.json`). 처음에는 좌표가 아니라 "구역 A → 구역 B로 직진 4초, 우회전 1.2초" 같은 상대 이동 시퀀스로 충분합니다.
3. **라벨 기반 임무 할당**: `mission.py`/`llm_control_node`가 "주방 순찰해", "복도 점검해" 같은 명령의 지역명을 위상 지도의 라벨과 매칭해서 해당 구역까지의 이동 시퀀스를 재생하도록 `patrol_node`를 확장합니다.
4. **구역 태깅된 이상 알림**: 5단계(이미 구현된 위험 알림)에 현재 위치의 지역 라벨을 붙여서 "복도에서 이상 감지: ..." 처럼 어디서 발생했는지 알 수 있게 합니다.

이 4가지는 각각 별도로 구현 가능한 단위 작업입니다. 다음에 진행할 때는 1번(탐색 모드 + 지역 분류 프롬프트)부터 시작하는 것을 권장합니다 — 나머지 단계가 전부 이 위에 쌓이기 때문입니다.

## 관련 문서
